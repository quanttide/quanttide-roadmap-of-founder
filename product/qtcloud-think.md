# 量潮思考云蓝图

## 产品定位

思考云是一个 System Two 觉察工具，核心定位是：

- 观察思考轨迹，识别思考状态
- 作为一面镜子，帮助用户校正认知，而非直接提升思考能力
- 相信用户自己会调整，系统只提供观察和反馈
- 从"镜子"进化为"认知审计系统"，提供思维模式的资产负债表

## 核心设计

### 原子定义：想法容器

1. 想法的原子定义 — 最小的"想法单元"包含：身份、时间、内容、来源、状态
2. 不可篡改性 — 想法一旦生成不可修改，只能追加备注，借鉴会计原始凭证原则
3. 容器连接规则 — 关系类型的最小集合：父子、引用、矛盾、演化
4. 生命周期模型 — 原始捕捉 → 待处理 → 已归档 / 已废弃 / 已转化为行动
5. 约束边界 — 用户主权与自动化的红线：不自动删除、未经同意不合并
6. 成功标准 — 降低想法丢失率、减少重复思考成本、提升回溯速度

### 认知审计框架

借鉴会计学建立结构化框架：

- 想法 = 原始凭证：一旦生成，不可修改，只能追加备注
- 卡片 = 明细账：机器可读的结构化数据
- 报告 = 财务报表：给决策者（用户）看的现状分析
- 试算平衡：矛盾关系即认知的"借贷不平"，系统检测到矛盾时触发认知审计
- 认知审计日志：用户不仅能看到想法，还能看到思维模式的资产负债表——哪些是资产（长期价值的沉淀），哪些是负债（未处理的矛盾、重复的焦虑）

### 六大功能

- 捕捉：任何时刻、最低摩擦接收想法，不评判、不丢失
- 存储：持久化保存想法及元数据，可回溯、可审计，引入事件溯源机制
- 关联：自动/半自动发现想法间的潜在连接，呈现思维图谱
- 约束执行：强制执用户设定的边界（不自动删除、债务显式化）
- 沉淀：将有长期价值的想法迁移到有序形态（知识库、行动项、债务卡片）
- 遗忘：安全丢弃不再有价值的想法，避免容器无限膨胀

## 数据处理流程

原始 message → 卡片（机器可读） → 日志 → 报告（格式化）

- 原始格式多样化（聊天记录等）
- 卡片更机器可读，由机器处理，用户无需直接交互
- 报告相对格式化，代表思考轨迹的演变
- 认知有连续性，迭代不会白干

### 报告即界面

- 用户唯一的交互界面是"报告"和"捕捉入口"
- 用户在阅读报告时产生的反馈（修正、连接）作为新的 Message 再次进入流程，形成闭环
- 不要让用户整理卡片，那是机器的事

### 分层处理策略

- L1 极速：本地规则处理（打时间戳、归档、去重），无需 AI
- L2 轻量 AI：本地小模型（7B以内），处理分类、提取实体、简单情感倾向
- L3 重量 AI：云端大模型，仅处理报告生成、深度矛盾检测、复杂演化路径
- 异步批处理：用户捕捉时只做存储，关联分析在后台批处理（如每4小时一次），避免实时调用的延迟和成本，同时给用户沉淀的时间间隙

## 关键洞察

### 健康认知状态

- 核心是作为镜子校正，定义"健康认知状态"是难点
- 批判性思维的基础：可溯源、可交叉验证
- 反复出现的模式可能是特点，看不到的则认为不是

### 与写作系统的联动

- 思考云单独作用有限，必须与写作系统搭配
- 思考报告是启发器，启发用户讲述思考故事
- 故事成为写作的原材料，思考与写作联动才能发挥最大效果

### 开发范式

- 本地模型作为校验器，验证开发逻辑是否合理
- 在本地模型执行范围内的逻辑，通常是模型无关的可靠逻辑
- 用 gallery 保存关键提示词和关键报告作为参考
- 避免不断复杂化平台逻辑

## 遗忘机制设计

冷数据归档：不真正删除，而是放入"冷库"

熵增管理：系统必须有"排熵"机制。如果一个想法长期未被关联、未被唤醒、且带有负面情绪标签，系统应建议用户举行"认知葬礼"——正式地标记它为废弃，并总结为什么它会变得不重要。这个过程本身就是一种认知校正。

半衰期概念：创造性想法的半衰期长，维护性压力的半衰期短。系统根据时间衰减连接权重，实现自然遗忘，而非硬性删除。

## 数据演化与版本化

事件溯源：不存储"最终状态"，存储"变更事件"。当分类逻辑改变时，重放历史事件即可得到新的归档结果。

版本化认知：保留不同时期的分类逻辑。认知是演化的，去年的"错误分类"也是当时认知状态的如实记录，这本身就是"思考轨迹"的一部分。

## 验证方向

### TUI 原型最小闭环

1. 输入：用户输入一段混乱的语音/文字
2. 镜子：系统立刻反馈一个"结构化的机器卡片"，验证原子定义是否完备
3. 回溯：一周后，系统能否准确复现当时的思考情境，验证时间与状态捕捉

### 想法分类

- 想法分类习惯（创造性 vs 维护性压力），提供参考分类
- 双钻模型变体：捕捉/存储 = 发散，关联/约束 = 收敛，沉淀 = 交付，遗忘 = 休整

## 困惑与挑战

1. 如何结构化地将认知工程思想放入程序，缺乏像财务那样明确的框架（已通过认知审计框架部分解决）
2. AI 执行效率不够高，开多个又容易管理不过来，需要仔细权衡（已通过分层处理和异步批处理缓解）
3. 数据结果变更可能导致归档站混乱（已通过事件溯源和版本化认知解决，在迭代中持续验证）
